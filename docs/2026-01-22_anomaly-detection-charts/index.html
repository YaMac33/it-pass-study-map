異常検出に最適な図：管理図で品質のバラつきを見抜く

  • #異常検出
  • #管理図
  • #品質管理
  • #原因分析
  • #プログラム

システム開発や品質管理の現場では、目的に応じて様々なグラフ（QC7つ道具）を使い分けます。中でも試験でよく問われるのが、「異常を検出したいとき」に使う図の選び方です。

結論から言うと、「限界線を超えた異常を見つける＝管理図」です。なぜ管理図が正解で、他が違うのか。それぞれの図の「形」と「目的」を整理して解説します。

1. 【正解】管理図 (Control Chart)

管理図は、工程が安定しているか（異常がないか）を時系列で監視するための図です。

管理図のイメージ

点線の「管理限界線（UCL / LCL）」を超えた点（赤色）があれば、
「異常が発生している」と即座に判断できます。

なぜこれが「異常検出」に最適なのか？

  • 基準が明確：「ここを超えたらアウト」という線（限界線）があらかじめ引かれているため。
  • 時系列：プログラムA、B、C...と順番に並べて、突発的なバグ発生（スパイク）を検知できるため。

2. その他の選択肢（間違いの理由）

試験では、管理図以外の図が「ひっかけ」として登場します。これらは「異常を見つける」のではなく、別の目的で使われます。

特性要因図 目的：原因分析

「なぜバグが起きたか？」の原因を深掘りする図。異常を見つけた後に使う。

パレート図 目的：優先順位

「どのバグが一番多いか？」を知る図。多い順に対策するために使う。

レーダーチャート 目的：特性比較

「性能・保守性・価格」など複数の項目のバランスを見る図。

試験対策：キーワードで即答する

問題文に以下のキーワードが出たら、対応する図を反射的に選べるようにしましょう。

キーワード 正解の図
限界線、異常検出、工程の安定 管理図
原因、魚の骨、因果関係 特性要因図
重点管理、2:8の法則、優先順位 パレート図
バランス、特性比較 レーダーチャート

まとめ

「限界線を超えたら異常」。このシンプルなルールを可視化してくれるのが管理図です。

試験問題で「上限・下限」や「限界線」という言葉を見つけたら、迷わず管理図を選んでください。
