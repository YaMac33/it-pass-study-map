カーディナリティの解説

  • #カーディナリティ
  • #データベース
  • #集合論
  • #ER図

「カーディナリティ（Cardinality）」とは、集合やデータの「要素の数」や「関係の多さ」を表す概念です。ITパスポートや基本情報技術者試験、そして実際のデータベース設計において頻出する重要用語です。

この記事では、集合論における基本的な意味と、データベース設計（ER図）における実務的な意味の2つの側面から解説します。

💡 この記事の要点
  • 集合論では「要素の数」を指す
  • データベースではテーブル間の「1対1」「1対多」「多対多」の関係性を指す
  • 「カーディナリティ ＝ 数の大きさ・関係の多さ」と覚える

1. 集合におけるカーディナリティ

数学の集合論において、カーディナリティはシンプルに「集合に含まれる要素の個数」を意味します。

集合 $A$ の要素数を $|A|$ と表します。

集合 A = { 1, 2, 3 }
📦 📦 📦
カーディナリティ |A| = 3
集合 B = { "リンゴ", "バナナ" }
🍎 🍌
カーディナリティ |B| = 2

この概念は、プログラミングにおいて配列やリストの長さ（Length/Size）を考える際の基礎となります。

2. データベースにおけるカーディナリティ

ITの実務、特にリレーショナルデータベース（RDB）の設計において、カーディナリティは「テーブル間のリレーション（関係）におけるレコードの対応数」を指します。

主に以下の3つのパターンに分類されます。

2.1 1対1（One-to-One）

テーブルAの1行に対して、テーブルBの1行のみが対応する関係です。

  • 例: 人とパスポート、社員と社員証
  • 特徴: 左右のテーブルの行数はほぼ同じになり、片方の行は必ずもう片方に対応します（あるいはNULL）。
[ テーブル: Person ]      [ テーブル: Passport ]
+----+-------+          +----+-----------+
| ID | Name  |          | ID | Number    |
+----+-------+          +----+-----------+
| 1  | Alice | <------> | 1  | MZ123456  |
| 2  | Bob   | <------> | 2  | MZ987654  |
+----+-------+          +----+-----------+

2.2 1対多（One-to-Many）

テーブルA（親）の1行に対して、テーブルB（子）の複数行が対応する関係です。最も頻繁に登場するリレーションです。

  • 例: 会社と社員、親と子、注文と注文明細
  • 特徴: 子テーブル側に、親テーブルのID（外部キー）を持ちます。
会社 (1)
雇用 ──────▶
社員 (多)
1つの会社には、複数の社員が所属する
[ テーブル: Company ]        [ テーブル: Employee ]
+----+-------+            +----+-------+------------+
| ID | Name  |            | ID | Name  | Company_ID |
+----+-------+            +----+-------+------------+
| 1  | TechA | <--------+ | 10 | John  | 1          |
| 2  | BizB  |       +--+ | 11 | Paul  | 1          |
+----+-------+       |    | 12 | Mary  | 2          |
                     +--- | 13 | Ken   | 1          |
                          +----+-------+------------+

2.3 多対多（Many-to-Many）

テーブルAの1行がテーブルBの複数行に対応し、かつテーブルBの1行もテーブルAの複数行に対応する関係です。

  • 例: 学生と授業（1人の学生は複数の授業を受け、1つの授業には複数の学生がいる）
  • 実装: リレーショナルデータベースでは直接表現できないため、間に中間テーブルを挟んで「1対多」の関係を2つ作ることで解決します。
[ Student ]       [ 中間: Student_Course ]      [ Course ]
+----+      +------------+-----------+      +----+
| ID |      | Student_ID | Course_ID |      | ID |
+----+      +------------+-----------+      +----+
| 1  | <--- | 1          | 100       | ---> | 100|
| 2  |      | 1          | 101       | ---> | 101|
            | 2          | 100       |
            +------------+-----------+

3. ER図風カーディナリティまとめ

システム設計図（ER図）では、これらの関係を線と記号で表現します。実務でよく使われる表記（IE記法など）の概念をテキスト図で整理しました。

ER図を読むコツ
左右のテーブルを結ぶ線の端にある記号（「1」や「∞（鳥の足）」）を確認します。

一目でわかる比較表

① 1対1

A ── 1 ── 1 ── B

② 1対多（親と子）

A ── 1 ── ∞ ── B

③ 多対多（中間テーブルが必要）

A ── ∞ ── ∞ ── B

まとめ

カーディナリティを理解することは、正しいデータベース設計を行うための第一歩です。

  • 集合論：要素の数（配列の長さなど）
  • DB設計：テーブル間のレコードの対応数（1:1, 1:N, N:M）

特に「1対多」の関係はアプリケーション開発で頻出するため、どちらが「1」でどちらが「多」なのかを常に意識して設計図を読み書きできるようにしましょう。
