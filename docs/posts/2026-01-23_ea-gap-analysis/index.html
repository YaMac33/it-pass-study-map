EAにおけるギャップ分析の基礎

  • #EA
  • #ギャップ分析
  • #業務プロセス
  • #As-Is
  • #To-Be

企業全体のシステムや業務を最適化する「エンタープライズアーキテクチャ（EA）」。
その計画段階で最も重要になるのが、現状と理想の差を明確にする「ギャップ分析」です。

1行要約
ギャップ分析とは、「今の姿（As-Is）」と「あるべき姿（To-Be）」を比較し、その差（ギャップ）を埋めるための課題を洗い出す手法のこと。

1. ギャップ分析の基本概念

EAでは、いきなり新しいシステムを作り始めることはありません。まず「今どうなっているか」と「どうなりたいか」を定義し、その間にある距離を測ります。

As-Is（現状） 現在の業務・システム
今の問題点
────────▶
Gap (差分)
To-Be（理想） 将来あるべき姿
目指すゴール

この「Gap」を埋める作業こそが、システム開発や業務改善プロジェクトになります。

用語の整理

  • As-Is（アズ・イズ）：現在の状態。「今のやり方」のこと。
  • To-Be（トゥー・ビー）：将来あるべき姿。「理想の状態」のこと。

「As-Is分析」といった場合、まずは現状の業務フローを可視化して問題点を洗い出す作業を指します。

2. 具体例：在庫管理システムの改善

抽象的な話だけでは分かりにくいので、「手作業だらけの在庫管理」を「自動化されたシステム」に変えるケースで見てみましょう。

EAの4つの視点（ビジネス、データ、アプリ、技術）などを意識しながら比較します。

業務
手作業で在庫チェック → 自動で在庫更新・発注
システム
Excel・紙台帳で管理 → 統合在庫管理システム
データ
店舗ごとにバラバラ → リアルタイム共有・一元化
組織
各店舗の担当任せ → 本社で一括管理

このように並べることで、「Excelをやめてシステムを入れる必要がある」「ネットワーク構築が必要だ」「店舗スタッフの教育が必要だ」といった具体的なアクションプラン（改善計画）が見えてきます。

3. 実施のステップ

実際のプロジェクトでは、以下の順序で進めます。

  1. As-Isの把握：今の業務フロー図を書く、システム構成図を作る。
  2. To-Beの定義：経営戦略に基づいた新しい業務フローやシステム像を描く。
  3. ギャップの特定：「何が足りないか」「何を変えるべきか」をリストアップする。
  4. 移行計画の策定：ギャップを埋めるためのスケジュールや予算を決める。

まとめ

EAやシステム刷新において、ギャップ分析は地図のような役割を果たします。

  • As-Is：現在地
  • To-Be：目的地
  • ギャップ：その距離とルート

試験では「現状分析（As-Is）を行い、理想（To-Be）との差（ギャップ）を明確にする」というフレーズがよく出題されるので、セットで覚えておきましょう。
