オブジェクト指向、SOA、マイクロサービスの進化の歴史と違い

  • #マイクロサービス

ソフトウェア開発の歴史は、「いかに部品化して再利用するか」の歴史でもあります。
「オブジェクト指向」から始まり、「SOA」を経て、現代の「マイクロサービス」へと至る設計思想の進化を整理します。

1行要約
「プログラムの中（オブジェクト指向）」から「システム同士（SOA）」へ、そして「クラウド上で独立（マイクロサービス）」へと、部品の単位が変化してきた。

2. 各アーキテクチャの詳細

① オブジェクト指向（Object Oriented）

1つのアプリケーションの中で、データと処理をセットにした「オブジェクト（クラス）」という単位で部品化します。
メリット： プログラムの再利用性が高まる。
限界： あくまで「1つのアプリの中」の話。システム間の連携は考えられていない。

② SOA（Service Oriented Architecture）

企業の基幹システムなどを「受注サービス」「在庫サービス」といった大きな業務機能単位で部品化し、ネットワーク越しに連携させます。
メリット： 既存のシステムを活かして統合できる。
限界： サービス単位が大きく、中央の連携基盤（ESB）が複雑になりがち。

③ マイクロサービス（Microservices）

SOAをさらに細かくし、クラウド技術（コンテナ等）を使って「1つの機能＝1つの独立したサービス」として作ります。
メリット： 一部だけ修正・デプロイできる。アクセス増に合わせて自動で拡張（スケーリング）できる。
課題： サービスが多すぎて管理が大変。

3. 比較まとめ

それぞれの違いを表で整理します。

項目 オブジェクト指向 SOA マイクロサービス
部品の単位 クラス 業務システム 小さな機能
通信 メモリ内呼出 ネットワーク (SOAP等) Web API (REST等)
主な舞台 アプリ開発 企業内システム統合 Webサービス・クラウド
メリット コードの保守性 資産の有効活用 変更への速さ・拡張性

まとめ

進化の歴史を一言で言うと、以下のようになります。

  • オブジェクト指向： プログラムをきれいに書くための技術。
  • SOA： バラバラな社内システムをつなぐための技術。
  • マイクロサービス： 変化の激しいネット社会で、素早くサービスを提供・改善するための技術。

「より独立して、より自由に動けるように」進化してきたと捉えましょう。
